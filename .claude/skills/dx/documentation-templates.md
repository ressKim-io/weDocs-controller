# Documentation Templates

문서 작성 시 아래 템플릿을 사용한다. 트리거 규칙은 `.claude/rules/documentation.md` 참조.

---

## ADR (Architecture Decision Record)

```markdown
# [기술/결정] 아키텍처 결정 (ADR)

작성일: YYYY-MM-DD
상태: Proposed | Accepted | Deprecated | Superseded by [NNNN]
프로젝트: {project-name} ({project-description})

---

## Decision Summary

| 결정 항목 | 선택 | 대안 | 핵심 근거 |
|-----------|------|------|-----------|

## 1. 배경 (Context)
## 2. 요구사항

| 요구사항 | 설명 | 중요도 |
|----------|------|--------|

## 3. 대안 비교

| 항목 | 선택지A | 선택지B | 선택지C |
|------|---------|---------|---------|

## 4. 결정 (Decision)
## 5. 결과 (Consequences)
### 긍정적 영향
### 부정적 영향 / 리스크
### 향후 과제
## 6. 참고 자료
```

### ADR 규칙
- 반드시 대안 비교표 포함
- 프로젝트 맥락에서 판단
- 트레이드오프 인정, 정량적 근거 우선
- 번호 순차 부여 (`docs/adr/` 최대 + 1)

## 프로젝트 개요서

```markdown
# {Project Name} — {Project Description}

최종 업데이트: YYYY-MM-DD

## 1. 프로젝트 소개 (비전, 핵심 기능, 기술적 도전)
## 2. 기술 스택 (레이어별 테이블 + ADR 링크)
## 3. 시스템 아키텍처 (High-Level)
## 4. 레포지토리 구조 (Repo, 역할, 배포 방식)
## 5. 팀 구성 및 역할
## 6. 마일스톤 / 로드맵
## 7. 관련 문서
```

## 포스트모템

```markdown
# [장애 제목] 포스트모템

발생일: YYYY-MM-DD HH:MM ~ HH:MM (KST)
심각도: SEV1 / SEV2 / SEV3

## 1. 요약 (영향, 지속 시간, 근본 원인)
## 2. 타임라인 (시간별 이벤트)
## 3. 근본 원인 분석
## 4. 영향 범위 (사용자 수, 실패 요청, 에러율, 데이터 유실)
## 5. 대응 과정
## 6. 재발 방지 조치 (Action Items 테이블)
## 7. 교훈 (잘한 점, 개선할 점, 행운)
```

## 마이그레이션 기록

```markdown
# [전환 주제] 마이그레이션 기록

상태: Planning | In Progress | Completed

## 1. 개요 (Before / After 테이블)
## 2. 왜 전환하는가
## 3. 전환 계획 (단계별 + 리스크/롤백)

## 5. Before / After 비교 (배포 시간, 리소스, 운영 복잡도)
## 6. 교훈
```

## 부하 테스트 결과

```markdown
# [시나리오명] 부하 테스트 결과

도구: K6 / Gatling / nGrinder

## 1. 테스트 목적
## 2. 테스트 환경 (인프라 + 앱 설정 + 도구 설정)
## 3. 테스트 시나리오
## 4. 테스트 결과
### 4.1 종합 요약 (p50/p90/p95/p99, 성공률, RPS, 에러율 — SLO 판정 필수)
### 4.2 엔드포인트별 상세
### 4.3 시간대별 추이
### 4.4 리소스 사용량
### 4.5 에러 분석
## 5. 병목 지점 분석
## 6. 개선 방안
## 7. 이전 테스트 대비
## 8. 첨부 (Grafana, K6 리포트, Pyroscope)
```

### 부하 테스트 규칙
- 모든 수치 실측값, 백분위수 필수
- 환경 정보 누락 금지, 시간대별 추이 기록
- 에러 분석 필수 (0건이어도 명시)
- SLO 판정 필수

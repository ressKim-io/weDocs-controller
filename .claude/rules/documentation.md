# Documentation Rules

프로젝트의 `docs/` 디렉토리에 프로젝트 문서를 작성할 때 반드시 따라야 할 규칙.
이 규칙은 문서 작성 트리거(시점)와 검증 규칙만 다룬다 — **템플릿 본문은 `/documentation-templates` skill 참조**.

## 자동 트리거 규칙 (MANDATORY)

Claude는 아래 상황을 감지하면 **사용자 요청 없이도** 해당 문서 생성을 제안하거나, 작업 완료 후 자동으로 작성한다.

### 즉시 작성 (작업 완료 직후 자동 생성)

| 트리거 상황 | 생성할 문서 | 조건 |
|------------|-----------|------|
| 기술 A vs B 비교 후 선택 | ADR | 대안이 2개 이상 있었던 기술 결정 |
| 부하 테스트 실행 완료 | 부하 테스트 결과 | K6/Gatling/nGrinder 결과가 나온 직후 |
| 장애 발생 → 원인 파악 → 수정 | 포스트모템 | 프로덕션/스테이징 환경 장애 |
| Compose→K8s, 모놀리스→MSA 등 전환 완료 | 마이그레이션 기록 | 환경/아키텍처 전환 작업 |
| Phase 게이트 통과 (SDD) | overview / migration / dev-log 동시 갱신 | Phase 단위 마이그레이션의 각 phase 완료 시 |
| CI/CD 파이프라인 신규 구축 또는 대폭 변경 | CI/CD 파이프라인 문서 | 워크플로우 파일 생성/수정 |
| Makefile, 스크립트, 도구 자동화 구축 | DX 개선 기록 | 개발자 워크플로우 개선 |

### 제안 후 작성 (사용자에게 먼저 물어봄)

| 트리거 상황 | 제안할 문서 | 질문 예시 |
|------------|-----------|----------|
| 인프라 구성이 크게 바뀜 | 인프라 아키텍처 업데이트 | "인프라 구성이 변경되었는데 아키텍처 문서를 업데이트할까요?" |
| 모니터링/알림 설정 작업 | SLI/SLO 정의서 | "SLI/SLO 문서를 작성하거나 업데이트할까요?" |
| 인프라 비용 관련 작업 (인스턴스 변경, 리소스 조정) | 비용 분석 | "비용 변경 사항을 FinOps 문서로 기록할까요?" |
| 스프린트/마일스톤 완료 | 프로젝트 회고 | "이번 마일스톤 회고를 작성할까요?" |
| 새 팀원 합류 또는 프로젝트 초기 설정 변경 | 온보딩 가이드 업데이트 | "온보딩 가이드를 업데이트할까요?" |

### 주기적 확인

| 주기 | 확인 사항 | 행동 |
|------|----------|------|
| 매 세션 시작 시 | `docs/project/overview.md` 존재 여부 | 없으면 생성 제안 |
| 인프라 작업 후 | 아키텍처 문서와 현재 상태 일치 여부 | 불일치 시 업데이트 제안 |
| ADR 작성 후 | `docs/adr/` 번호 연속성 | 번호 gap 없는지 확인 |

## Issue 생성 트리거 (MANDATORY)

`gh issue create` (또는 동등한 작업) 실행 전 반드시 아래 절차를 따른다:

1. 레포 루트의 `.github/ISSUE_TEMPLATE/` 디렉토리 확인 — 템플릿 파일 존재 여부, 형식(MD 또는 YAML form)
2. 해당 템플릿의 `body[].attributes.label` (YAML) 또는 섹션 헤더(MD) 구조 그대로 본문 작성
3. 템플릿이 `labels:` 필드를 정의하면 `--label` 플래그로 동일 라벨 지정
4. 레포마다 템플릿 종류/필드가 다름 — 다른 레포의 형식을 재사용하지 않는다

템플릿이 부재한 레포에서만 자유 형식 사용. 단, `## Summary`, `## Reproduction`, `## Acceptance Criteria` 같은 표준 헤더는 유지.

## SDD 작성 → 코드 리뷰 워크플로우 (MANDATORY)

Spec-Driven Development(SDD) 문서를 작성한 직후 다음 순서를 따른다:

1. SDD 문서 작성 완료
2. **`code-reviewer` 에이전트 호출** — SDD 자체를 리뷰 대상으로 (요구사항 누락, 일관성, 검증 가능성)
3. 리뷰 결과를 사용자에게 텍스트로 표시
4. 사용자 승인 후 구현 단계로 진입

**금지**: SDD 작성 직후 곧바로 코드 구현. SDD가 검토되지 않은 채 구현되면 변경 비용이 폭증한다.

## 작성 검증 규칙 (MANDATORY)

템플릿을 따라 작성하더라도 아래 검증 규칙을 충족해야 한다. 미충족 시 문서를 다시 작성한다.

### ADR 검증 규칙

- **반드시 대안 비교표 포함** — "X를 선택했다"만으로는 부족. 왜 Y, Z가 아닌지 설명
- **프로젝트 맥락에서 판단** — 일반론이 아닌 "이 프로젝트에서 왜 이것인가"
- **트레이드오프 인정** — 선택한 기술의 단점/리스크도 명시
- **정량적 근거 우선** — 벤치마크, 처리량, 리소스 사용량 등 수치 포함
- **번호 순차 부여** — `docs/adr/` 내 기존 최대 번호 + 1

### 부하 테스트 결과 검증 규칙

- **모든 수치는 실측값** — 추정치 사용 금지. 데이터 없으면 "미측정"
- **백분위수 필수** — 반드시 p50, p90, p95, p99 포함
- **환경 정보 누락 금지** — 재현 불가능한 결과는 무의미
- **시간대별 추이 기록** — VU 증가에 따른 변화를 보여야 병목 파악 가능
- **에러 분석 필수** — 에러가 0건이어도 "에러 없음" 명시
- **이전 결과 대비** — 반복 테스트 시 이전 결과와 비교
- **SLO 판정 필수** — 각 지표가 기준을 통과했는지 ✅/❌ 명시

### 포스트모템 검증 규칙

- **타임라인 필수** — 감지 / 조사 시작 / 임시 조치 / 근본 원인 파악 / 수정 배포 / 정상화
- **근본 원인 ≠ 증상** — "왜?"를 5번 묻기. workaround로 끝내지 않음
- **영향 범위 정량화** — 사용자 수, 실패 요청 수, 에러율 피크, 데이터 유실 여부
- **Action Items 우선순위 명시** — P0/P1/P2 + 담당 + 기한

### 마이그레이션 기록 / DX 개선 / 비용 분석 검증 규칙

- **Before / After 필수** — "빨라졌다" 금지, "p95가 500ms → 200ms로 60% 개선" 형태로 정량화
- **소요 시간/리소스/복잡도** 같은 측정 가능한 지표를 표로 비교

## 공통 작성 규칙

- **한국어로 작성** — 기술 용어는 원문 유지 (예: "Kafka", "p99")
- **테이블 적극 활용** — 비교, 수치는 산문보다 테이블이 명확
- **날짜 형식**: `YYYY-MM-DD` (ISO 8601)
- **상태 관리**: ADR은 `Proposed → Accepted → Deprecated/Superseded` 라이프사이클
- **상호 참조**: 관련 문서 간 상대 링크 사용 (예: `[Kafka ADR](../adr/0001-kafka-adoption.md)`)
- **Before/After 필수**: 개선 작업 문서는 반드시 이전/이후 비교 포함
- **정량적 데이터 우선**: "빨라졌다"가 아니라 "p95가 500ms → 200ms로 60% 개선"

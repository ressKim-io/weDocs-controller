# Professional Writing Rules

포트폴리오, 슬라이드, 이력서, README, 발표 자료 등 **자기 자신을 어필하는 한국어 카피**를 작성할 때 적용한다.
일반 코드 주석이나 기술 문서에는 적용하지 않는다.

---

## 절대 금지 (MANDATORY)

| 금지 표현 | 이유 |
|---------|------|
| "15인 팀", "팀 리더", "팀 규모 X명" | 팀 규모는 본인 능력을 증명하지 않음. 의미 없는 정보. |
| "200 skills", "50 agents", "20 rules" 같은 **자산 갯수 나열** | 갯수만으로는 능력/결과가 안 보임. AI 자산 카운트는 누구나 만들 수 있어 변별력 0. |
| "AI를 도구가 아닌 환경으로" 같은 **추상 슬로건** | 첫 인상에서 "이게 뭔데?" 로 넘김. 구체 사실/결과로 대체. |
| "DevOps 전 영역 단독 책임" 같은 **자기 자랑** | 사실 검증 안 됨. 변화량/결과로 대체. |
| 본인이 만든 비표준 표현 ("환원", "박는다" 등) | 외부 검색으로 통용성 검증 필수. 통용 안 되면 일상 단어로 풀어쓰기. |

---

## 반드시 적용

### 1. 첫 페이지 2분 룰

채용 담당자가 한 사람당 포트폴리오에 쓰는 시간은 2분 미만이다.

- 첫 페이지에서 "이 사람은 ○○을 할 수 있는 사람" 즉각 확신 가능해야 함
- 슬로건/철학은 본문/closing에서, **첫 페이지는 사실/결과만**
- 자기소개 슬라이드 = 직함 + 본인이 만든 결과 + 능력 키워드

### 2. "내가 할 수 있는 것" 중심

- 자산 소개: "X개 만들었다" ❌ → "X로 Y를 자동화했다" ⭕
- 갯수가 아닌 **자산이 만든 결과** 중심
  - ❌ "194 skills 운영"
  - ⭕ "89건 트러블이 자동으로 rule이 되는 사이클"

### 3. 숫자는 변화량으로

- ❌ "37개 / 380개" (단순 양)
- ⭕ "p50 189→18ms" / "5xx 10.88% → 0%" (Before/After 변화)
- 과장 금지 — 실제 검증한 수치만 사용 (50만 → 10만이면 10만으로 적기)

## 슬라이드 헤드라인 = 면접관과 대화하는 도구

슬라이드 텍스트는 **본인 발표를 받쳐주는 대화 도구**다. 면접관이 헤드라인을 보고 5초 안에:
1. 무엇에 대한 페이지인지 파악
2. 본인에게 **무엇을 묻고 싶어지는지** 자연스럽게 유도

### 좋은 헤드라인 (질문 유도형)

| 패턴 | 예시 |
|------|------|
| 질문형 | "10만 동시접속을 어떻게 흡수했나" — "어떻게?" 즉시 질문 |
| 도전 + 결과 암시 | "야구 티켓 10만 동접 — DevOps에서 풀었습니다" — "어디서?" |
| 변환 비교 | "AI를 쓴 게 아니라, 운영했습니다" — "차이가?" |

### 나쁜 헤드라인

| 패턴 | 예시 | 문제 |
|------|------|------|
| 추상 슬로건 | "AI를 도구가 아닌 환경으로" | "이게 뭔데?" |
| 자기 자랑 | "전 영역 단독 책임" | 사실 검증 없음 |
| 모호 동사 | "대규모 X, 3가지로 풀었습니다" | "뭘?"이 빠짐 |

## 한국어 자연스러움 7원칙

영어 직역/명사화/비교 암시 강조어가 가장 자주 보이는 어색함의 원인이다.

### 1. 명사화 → 동사화

한국어는 동사를 그대로 쓰는 게 자연스럽다.

- ❌ "흡수한 효과입니다" → ⭕ "흡수되는 것을 확인했습니다"
- ❌ "사용자 인증 처리가 완료된 후" → ⭕ "사용자를 인증한 후"

### 2. 한자어 동사 ("수행/실행/진행") 빼기

- ❌ "로그를 삭제하는 작업을 수행합니다" → ⭕ "로그를 삭제합니다"
- ❌ "모니터링을 실행하여" → ⭕ "모니터링으로"

### 3. 무생물 주어 회피

- ❌ "관측이 사건을 잡았습니다" → ⭕ "대시보드로 4건을 잡았습니다"
- ❌ "시스템이 결정합니다" → ⭕ "시스템에서 결정합니다"

### 4. "진짜/실제" 같은 비교 암시 강조어 회피

`진짜 X`라고 하면 자동으로 `가짜 X` 비교 대상을 암시 → 어색.

- ❌ "진짜 사건 4건을 잡았습니다" → ⭕ "운영 중 마주친 문제 4건"

### 5. 격식·모호 단어 → 구체 단어

| ❌ 모호 | ⭕ 구체 |
|---------|---------|
| 사건 | 문제 / 이상 / 장애 / 이슈 |
| 처리 | 발견 / 잡음 / 해결 / 막음 |
| 효과 | 결과 / 검증 / 확인 |

### 6. 헤드라인 ↔ 본문 메시지 일관성

헤드라인은 **본문이 증명하는 것을 요약**해야 한다. 모순 시 독자 혼란.

### 7. 도메인 시작 → 기술 결과

기술 수치만 박는 카피는 비기술자에게 "그래서 뭐?". **도메인 맥락 → 설계 의도 → 검증 결과** 흐름.

- ❌ "588K 중 390K가 Edge에서 처리되었습니다"
- ⭕ "티켓팅에서 사용자가 가장 많이 보내는 요청은 대기열 polling입니다. CDN Edge에서 받아내도록 설계했고, 588K 중 97%가 흡수되는 것을 확인했습니다."

## 직역 한자어 → 자연 한국어

| 직역 | 자연 한국어 |
|------|---------|
| 관측성 (Observability) | 모니터링 |
| 정량화 (Quantify) | 숫자로 봤습니다 |
| 의식적 (Conscious) | 일부러, 의도적 |
| 커스텀 (Custom) | 직접 구축, 맞춤 |

이공계 논문/매뉴얼에서는 흔하지만 **면접관 대화 톤에서는 어색**.

## 작성 후 자기 점검 4문항

1. **PDF로만 보는 사람도 이해되는가?** — 발표 보충 의존 금지
2. **헤드라인이 본문 증명할 것을 요약하는가?** — 둘 사이 모순 없나
3. **무생물 주어로 영어 직역하지 않았나?** — "X가 했다" 표현 점검
4. **"진짜/실제" 같은 강조어가 비교 대상을 암시하지 않나?**

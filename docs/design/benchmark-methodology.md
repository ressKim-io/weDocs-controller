# 벤치마크 방법론 — CRDT Engine

> **위상**: [엔진 설계서 §7](crdt-engine.md#7-벤치마크--n배-빠르다를-정직하게-만드는-법)의 확장. "무엇을 · 무엇 대비 · 어떻게" 측정해 **"N배 빠르다"를 정직한 주장으로** 만든다.
> **연결**: 가드레일 5(엔진=최적화+criterion 벤치) · [SDD §3.2](../sdd/2-services-and-contracts.md) · NFR(수렴 지연 p95\<100ms) · 코드 `weDocs-crdt-engine/benches/`.
> 작성: 2026-06-30

---

## 0. 문제 정의 — 지금 벤치는 왜 부족한가

현재 `benches/convergence.rs`:
```
bench: apply_256_concurrent_updates
  256개 "1글자 독립 삽입" update를 빈 Doc에 적용 → state_vector
  결과: 256 concurrent ≈ 1.166ms
```

세 가지 정직성 문제:

```
① baseline 없음        → "1.166ms"는 빠른가 느린가? 비교 대상 0 → "N배"의 N이 공허
② 측정 축 혼동         → criterion은 "머지 처리량", NFR은 "e2e 수렴 지연". 다른 것을 잼
③ 워크로드 비현실      → "256개 1글자 독립 삽입"은 실제 편집 분포 아님(순차 타이핑·삭제·붙여넣기 없음)
   + 측정 위생 결함     → Doc::new() 가 b.iter() 안 → alloc·decode가 머지 측정에 섞임 (§7)
```

이 문서가 ①②③를 푼다.

## 1. 측정 3계층 (가장 중요한 프레임)

**무엇을 재는지 헷갈리면 벤치가 거짓말이 된다.** 세 계층을 명시적으로 분리한다.

```
┌────────────────────────────────────────────────────────────────────┐
│ Tier 1  머지 마이크로벤치        도구: criterion                      │
│         "decode+apply 핫패스가 빠른가" — 결정론적·회귀 가드           │
├────────────────────────────────────────────────────────────────────┤
│ Tier 2  엔진 동시성/처리량       도구: criterion + tokio (or 커스텀)  │
│         "전역락 vs 샤딩, 다문서 동시 처리량" ← ★ "N배"의 N이 여기서   │
├────────────────────────────────────────────────────────────────────┤
│ Tier 3  e2e 수렴 지연           도구: 부하 하니스(별도, criterion 아님)│
│         "키 입력 → 타 클라 반영까지 p95<100ms" ← NFR 검증            │
└────────────────────────────────────────────────────────────────────┘
```

| Tier | 답하는 질문 | 대표 지표 | 뒷받침하는 주장 |
|---|---|---|---|
| **1** | 머지 코어가 효율적인가 | ns/op, ops/sec | "yrs 핫패스 회귀 없음" |
| **2** | 동시 다문서에서 안 막히나 | ops/sec, 락 대기 p95 | **"샤딩으로 N배"**(설계서 §6) |
| **3** | 사용자 체감 지연 | 수렴 지연 p50/p95/p99 | **NFR p95\<100ms** |

> 면접 함정 회피: "criterion으로 1.166ms 나왔어요"는 Tier 1. "그래서 사용자가 빠른가?"는 Tier 3. **둘을 섞어 말하면 안 된다.**

## 2. Baseline 정의 — "N배"를 정직하게

"N배 빠르다"가 성립하려면 **비교 대상이 공정**해야 한다.

```
⛔ 반칙: "Java 게이트웨이는 Y.Doc이 없어서 못 함" → N배
   = 비교 불가는 비교가 아니다. (게이트웨이는 애초에 머지를 안 함)

✅ 정직한 baseline 후보:
   B1. full-state 동기화  vs  state-vector diff 동기화   → 전송 바이트·지연
   B2. 전역 Mutex 레지스트리  vs  per-doc 샤딩(DashMap)   → 다문서 동시 처리량  ★1순위
   B3. v1 인코딩/디코딩 오버헤드 자체 측정                 → 절대 수치의 맥락 제공
```

**권장 = B2를 주력으로.** 이유:
- "엔진 깊이"(설계서 §6 최적화 로드맵)와 직결 — 전역락 제거가 곧 N배 근거.
- **before/after 자가 비교**라 공정(같은 코드베이스, 같은 워크로드, 변수 하나).
- 면접에서 "엔진을 어떻게 최적화했나"에 수치로 답 가능.

> B1은 "왜 diff 동기화인가"를 정량화(설계서 §3.2). B3은 "lib0 v1이 비싼가"의 맥락. 셋 다 갖추되 **헤드라인 숫자는 B2**.

## 3. 워크로드 모델 — 대표성

현재 "256개 1글자 독립 삽입"은 비현실적. 실제 편집은:
```
순차 타이핑    같은 client가 연속 삽입 (clock 증가) — 가장 흔함
삭제          tombstone 생성 (CRDT 세금, 설계서 §3.5)
큰 붙여넣기    한 update에 수백~수천 char
late-join     신규 클라가 큰 문서에 state-vector diff로 진입
```

**워크로드 파라미터화**: `문서 수 D × 문서당 세션 S × update 크기 × (삽입:삭제 비율)`

대표 시나리오 3종(Tier 2/3 공용):
| 시나리오 | D × S | 무엇을 드러내나 |
|---|---|---|
| **핫 도큐먼트** | 1 doc × 50 session | fan-out 부하·broadcast cap(NFR 동시 50명) |
| **다문서** | 1000 doc × 2 session | **전역락 병목**(B2의 무대) |
| **late-join** | 큰 doc 1개 + 신규 진입 | diff 동기화 효율(B1) |

## 4. Tier 1 — 머지 마이크로벤치 개선

현재 단일 케이스를 **그룹 + baseline + 측정 위생**으로.

**측정 위생 수정(즉시):**
```
현재: b.iter(|| { let doc = Doc::new(); for u in &updates { decode+apply } })
      → alloc + decode 가 머지 측정에 혼입

개선:
  - "순수 머지" 재려면 → update를 iter 밖에서 미리 decode, iter 안엔 apply만
  - "doc 구성 비용" 재려면 → 현재대로 두되 케이스 이름을 build_doc_from_updates로 명확히
  → 두 케이스를 분리해 decode와 merge 비용을 각각 본다
```

**추가 케이스(워크로드 대표성):**
```
group "merge"
  ├─ sequential_typing_256     순차(같은 client) — 실제 타이핑
  ├─ concurrent_inserts_256    동시(독립 client) — 현재 케이스
  ├─ with_deletes_50pct        삭제 포함 — tombstone 경로
  └─ large_paste_10k           큰 update 1개 — 배치 머지
```

**회귀 가드(CI):**
```sh
cargo bench -- --save-baseline main      # main에서 기준 저장
cargo bench -- --baseline main           # PR에서 비교 → % 회귀 감지
```
가드레일 5("엔진")를 CI로 강제: 임계 초과 회귀 = 실패.

## 5. Tier 2 — 동시성/샤딩 처리량 (★ N배의 무대)

**핵심 실험: 전역 Mutex vs per-doc 샤딩, 같은 워크로드.**

```
워크로드: 다문서 시나리오(1000 doc × 2 session), tokio 멀티스레드 런타임
         N개 태스크가 동시에 apply_v1(서로 다른 doc) 호출

측정 A) 현재 Arc<Mutex<HashMap>> → 모든 apply가 전역 락 직렬화
측정 B) DashMap/샤딩 → doc별 독립 락

지표: 총 ops/sec, 락 획득 대기 p50/p95/p99, 코어 스케일링(1→2→4→8 thread)
```

```
기대 그래프 (가설):
ops/sec │           ╭─ B: per-doc 샤딩 (코어 따라 선형 증가)
        │        ╭──╯
        │     ╭──╯
        │  ╭──╯
        │ ╱─────────── A: 전역 락 (1코어 처리량에 포화)
        └──────────────────▶ thread 수
          ↑ 이 격차가 "N배"의 N (정직한 baseline=B2)
```

> 주의: "같은 doc = 같은 인스턴스"(consistent-hash)라도 **한 인스턴스가 수천 doc**를 들면 전역락이 병목. 그래서 이 실험이 실전 의미가 있음(설계서 부록 A-1).

## 6. Tier 3 — e2e 수렴 지연 (NFR 검증, criterion 아님)

criterion은 인프로세스라 네트워크·게이트웨이·gRPC를 못 잼 → **별도 부하 하니스** 필요.

```
하니스 구성:
  N개 가짜 클라(y-websocket 와이어 or yrs 클라)
       │ WS
       ▼
  ws-gateway(실기동) ──gRPC──▶ engine(실기동)
       │
  측정: 클라 A가 update 송신한 wall-clock T0
        → 클라 B가 그 update 수신·적용한 T1
        수렴 지연 = T1 - T0,  p50/p95/p99 집계
  동시 검증: 모든 클라 최종 상태 동일(수렴 정확성)
```

- 목표: **p95 \< 100ms**(NFR, 로컬 sync 기준).
- 도구 후보: 경량 Rust/JS 하니스(yrs 클라 N개) 또는 `k6` + ws 확장. 부하 곡선(동시 세션 10→50→100)으로 NFR 한계 탐색.
- 측정 환경 고정·기록(머신·네트워크·커밋 해시).

## 7. 측정 위생 체크리스트 (criterion 함정)

```
☑ black_box로 dead-code 최적화 차단              (현재 ✅)
☐ setup(alloc/decode)를 iter 밖으로 — 측정 대상만 iter 안에   (현재 ✗, §4)
☐ 워밍업·샘플 수 명시(criterion 기본 100 샘플, warm-up 3s)
☐ CPU governor=performance, 백그라운드 부하 제거
☐ --save-baseline 으로 절대 비교 가능하게
☐ 측정 머신·OS·커밋을 dev-log에 기록(재현성)
☐ throughput(Elements/byte) 모드로 ops/sec·MB/s 직접 보고
```

## 8. 결과 → 주장 매핑 (기록 규약)

벤치는 숫자만 내면 의미 없음 — **어떤 주장을 뒷받침하는지** 같이 기록.

| 벤치 | Tier | 뒷받침 주장 | 기록 위치 |
|---|---|---|---|
| merge group | 1 | "핫패스 회귀 없음" | criterion HTML + dev-log |
| 전역락 vs 샤딩 | 2 | **"다문서 N배"**(설계서 §6) | dev-log(환경·N·그래프) |
| e2e 수렴 지연 | 3 | **"NFR p95\<100ms 충족"** | dev-log + DoD 트래커 |

> 규약: 새 벤치 수치는 반드시 (a)환경 (b)baseline (c)뒷받침 주장 3종을 dev-log에 같이 남긴다. 숫자만 있는 벤치는 "정직하지 않은 N배"의 씨앗.

## 9. 로드맵

```
즉시(M1.5)  Tier 1 위생 수정 + 그룹화 + --save-baseline CI 회귀 가드
M2/M3      Tier 2 샤딩 before/after (전역락 제거와 함께) → "N배" 헤드라인 확보
M5(쇼케이스) Tier 3 부하 하니스 → NFR p95<100ms 실측 + 폴리글랏 trace 위에 지연 시각화
```

## 부록 — DevOps/엔지니어링 관점 자문

1. **Tier 1과 Tier 3은 상관이 약할 수 있다.** 머지가 2배 빨라져도 e2e 지연은 네트워크·게이트웨이 번역이 지배하면 거의 안 변함. 그럼 "머지 최적화"의 사업적 가치를 어떻게 정당화할까? (답 힌트: 머지는 처리량 한계 = 동시 문서 수 상한을 정함 → 비용/스케일 축에서 의미)
2. **샤딩 N배(Tier 2)를 보여도, consistent-hash로 "doc당 1인스턴스"면 한 인스턴스의 다문서 부하가 실제로 얼마나 높을지**가 전제다. 홈랩 규모(NFR: 수천 커넥션)에서 그 전제가 성립할 워크로드를 어떻게 추정할까?
3. **벤치를 CI 회귀 가드로 넣으면**, 노이즈 많은 CI 러너에서 false positive가 난다. criterion 회귀 임계를 어떻게 잡아야 "진짜 회귀"만 잡을까(통계적 유의성 vs 절대 %)?

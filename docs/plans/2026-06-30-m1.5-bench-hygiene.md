> 코드는 **서비스 레포 `weDocs-crdt-engine`** 에서 작업(feature 브랜치 + PR + 건별 승인).

## Context

[벤치마크 방법론](../design/benchmark-methodology.md) §0 이 진단한 현재 벤치(`benches/convergence.rs`)의
**측정 위생 결함**을 고친다. 핵심 문제:

1. `Doc::new()` + `Update::decode_v1` 가 `b.iter()` **안**에 있어 **alloc·decode 비용이 머지 측정에 혼입**
   ([§7](../design/benchmark-methodology.md#7-측정-위생-체크리스트-criterion-함정) "setup을 iter 밖으로" 위반).
2. 단일 케이스(`256개 1글자 독립 삽입`)뿐 → **워크로드 비대표성**(순차 타이핑·삭제·붙여넣기 없음, §3).
3. baseline 비교 장치 없음 → "N배" 회귀 추적 불가(§4 회귀 가드).

이번 PR(방법론 §9 로드맵 "즉시(M1.5)" 항목)은 **Tier1 머지 마이크로벤치**만 정직하게 만든다.
Tier2 샤딩(전역락 제거)·Tier3 e2e 하니스는 범위 밖(M2/M3/M5).

검증 완료(resolved 버전 기준, deep-thinking/config-contract-audit):

  `Throughput::ElementsAndBytes{elements,bytes}` 존재 확인(docs.rs/0.8.2).
- yrs **0.27.2**: `Text::push(txn,&str)`, `Text::remove_range(txn,index:u32,len:u32)`,
  `Update::decode_v1`, `apply_update(update)`(by value) — 캐시 소스 확인.
- 엔진 레포 `.github/workflows` **부재** → 회귀 가드는 Makefile 타깃 + README 문서로 도입(CI 신설은 범위 밖, 사용자 확정).

## 설계 — `benches/convergence.rs` 재구성

### 워크로드 생성기(생성은 타이밍 밖, §3 대표성)
- `concurrent_inserts(256)` — 기존 `make_updates`(독립 client 1글자 256개). 현재 케이스 보존.
- `sequential_typing(256)` — **같은 client** 연속 삽입. 각 update = 직전 state_vector 대비 diff(1글자).
- `with_deletes(256)` — 256 insert + 128 단일문자 delete(50% tombstone). `remove_range` 경로.
- `large_paste(10_000)` — 1만 글자 **단일 update 1개**(배치 머지).

### 측정 위생 — 2 그룹으로 decode/merge 비용 분리(§4)
- **group `merge`** (헤드라인, 순수 머지): `iter_batched` 로 `Doc::new()` + `decode_all()` 을
  **setup(타이밍 제외)** 에 두고, routine 은 `apply_update` 루프 + 최종 `state_vector` black_box 만.
- **group `build_doc`** (전체 파이프라인): 기존 `b.iter()` 패턴(alloc+decode+merge).
  `build_doc - merge` 델타 = decode+alloc 비용을 케이스별로 가시화.
- 두 그룹 모두 4개 워크로드 동일 적용(대칭).
- **per-update 1 transaction 유지** — 실서비스 `service.rs::apply_v1`(프레임당 1 apply)과 동일 경로 대표성.
- `throughput(Throughput::ElementsAndBytes{elements, bytes})` → ops/sec + MiB/s 동시 보고(§7).

### 회귀 가드 — Makefile 타깃 + README(§4, CI 신설 없이)
```makefile

	cargo bench -- --baseline main
```

## 검증

```sh
cd weDocs-crdt-engine && make proto-sync
cargo fmt --check && cargo clippy --benches -- -D warnings
cargo bench                            # 8 케이스(merge/build_doc × 4)
cargo bench -- --save-baseline main    # baseline 저장
cargo bench -- --baseline main         # 비교 모드
```
정상: 8 케이스 측정 출력, `merge/*` < `build_doc/*`(decode 제외 효과). 수렴 정확성은 기존 proptest 가드.

## 4. 핵심 기능 (MVP scope)

### 4.1 실시간 협업 편집
- 다중 사용자 동시 텍스트 편집, 충돌 없는 수렴(CRDT)
- 실시간 presence: 타 사용자 커서 위치·선택 영역 표시 (휘발성)
- 문서 영속화: 새로고침·재접속 후 상태 복원

### 4.2 AI co-pilot (읽기 전용 보조)
- **RAG 문서 Q&A**: 문서 내용을 임베딩·검색해 근거 기반 답변, 토큰 스트리밍(SSE)
- **요약/리라이트**: 선택 텍스트 또는 전체 문서를 요약·재작성, 결과는 **사용자가 명시적으로 삽입**(AI는 문서를 직접 수정하지 않음)
- LLM 소스: 로컬 Ollama 우선, 폴백 시 클라우드(OpenAI/Anthropic 등)

### 4.3 문서/권한 관리
- 문서 생성/조회/삭제
- 권한 모델: **비공개 + 멤버 초대** (1차 범위)
- 협업 룸 lifecycle (입장/퇴장/세션)

### 4.4 인증
- JWT 기반 인증, WebSocket 핸드셰이크 시 검증
- (MVP) 인증 로직은 Doc/Session 서비스에 통합, 추후 분리 가능

## 5. 명시적 비범위 (Out of Scope, 1차)

설계 비대화를 막기 위해 **의도적으로 제외**한다. README에 "확장 로드맵"으로 표기.

- ❌ 오프라인 편집 (CRDT가 가능케 하지만 동기화 복잡도 ↑ → 2차)
- ❌ 화이트보드/캔버스 협업 (텍스트에 집중)
- ❌ AI 이어쓰기·자동완성 (AI가 문서를 직접 편집하지 않는 단순성 유지)
- ❌ AI 편집 제안(suggest edits) — 위와 동일 이유
- ❌ 공개 문서·링크 공유·실시간 댓글 (멤버 초대 권한만)
- ❌ 멀티클라우드 failover (별도 프로젝트에서 이미 증명, 생략)

## 6. 비기능 요구사항 (NFR)

| 항목 | 목표 (1차) | 비고 |
|---|---|---|
| 동시 편집자 / 문서 | ~50명 | CRDT·presence fan-out 검증 충분 |
| 동시 WebSocket 커넥션 | ~수천 (홈랩 규모) | Java Virtual Thread 게이트웨이 검증 |
| 편집 반영 지연(로컬 sync) | p95 < 100ms | 로컬-우선 UI라 체감 0ms, 서버 반영 기준 |
| AI 응답 시작 지연(SSE 첫 토큰) | p95 < 5s (로컬 추론) | GPU 큐 대기 + 추론 |
| CRDT 수렴 | 100% (property-based 검증) | 어떤 머지 순서에도 동일 결과 |
| 보안 | 서비스 간 mTLS(Istio), JWT 인증 | |
| 관찰성 | 폴리글랏 단일 trace, 핵심 메트릭·로그 | OTel |

## 7. 성공 기준 (Definition of Done)

- [ ] 두 브라우저에서 같은 문서를 동시 편집해 충돌 없이 수렴한다
- [ ] 재접속 후 문서 상태가 복원된다
- [ ] 타 사용자 커서·선택이 실시간 표시된다
- [ ] 문서 근거 기반 AI 답변이 스트리밍으로 출력된다
- [ ] 로컬 GPU 과부하 시 클라우드로 폴백된다
- [ ] Java→Rust→Python 호출이 단일 trace로 관측된다
- [ ] K8s(홈랩)에 GitOps로 배포된다
- [ ] CRDT 수렴이 property-based 테스트로 검증된다
- [ ] README에 아키텍처 결정(언어 분리 근거 포함)이 문서화된다

## 8. 핵심 가정 / 리스크

- **가정**: 클라이언트 Yjs ↔ 서버 yrs(Rust)가 동일 binary wire format으로 상호운용된다 → M1에서 최우선 검증
- **리스크(상)**: Rust/CRDT 학습 곡선 → M1에 리스크 집중 배치(가장 먼저 검증)
- **리스크(중)**: 폴리글랏 빌드/CI 복잡도 → 5-repo 폴리레포 + buf submodule + controller 오케스트레이션으로 완화
- **리스크(중)**: 단일 GPU 동시 추론 한계 → 추론 큐 + 폴백을 설계에 내장

## 1. 정보구조 (IA)

```
Workspace "우리 팀"
├── 📄 온보딩
│   ├── 📄 첫날 체크리스트
│   └── 📄 개발환경 셋업
├── 📄 엔지니어링
│   ├── 📄 백엔드 가이드
│   │   └── 📄 DB 스키마          ← 페이지 안에 페이지 (무한 중첩)
│   └── 📄 배포 런북
└── 📄 회의록
    └── 📄 2026-06-30 스프린트
```

세 단계 컨테이너:
- **Workspace** — 팀의 지식 경계. MLP는 1개로 시작(데이터는 N 지원, D-6).
- **Page** — 위키의 한 글 = 한 CRDT 문서. 동시에 **자식 페이지의 부모**가 될 수 있다(트리).
- **Block** — 페이지 내용을 이루는 최소 단위(아래 §2).

> 사이드바 = 페이지 트리(접기/펼치기). 메인 = 선택한 페이지의 블록 에디터. 노션과 동일한 2-pane 멘탈모델.

```
┌────────────┬───────────────────────────────────┐
│ 사이드바    │  📄 백엔드 가이드                   │
│ (트리)     │  ─────────────────────────────    │
│            │  # 개요                            │
│ ▾ 온보딩    │  우리 백엔드는 …                    │
│   첫날…     │                                   │
│ ▾ 엔지니어링 │  - [ ] 할 일 1                     │
│   ▸ 백엔드   │  - [x] 할 일 2                     │
│   배포…     │                                   │
│            │  ```rust                           │
│ ＋ 새 페이지 │  fn main() {}                      │
│            │  ```          👤Alice  👤Bob(편집중) │
└────────────┴───────────────────────────────────┘
```

## 2. 블록 에디터 스펙

페이지 내용 = **블록의 순서 있는 트리**. 각 블록은 타입 + 내용. (Tiptap이 이미 이 모델 → Yjs `Y.XmlFragment`로 CRDT 매핑)

**MLP 블록 타입 (최소 위키 작성 가능 세트):**

| 타입 | 용도 | 비고 |
|---|---|---|
| paragraph | 본문 | 기본 |
| heading 1–3 | 구조 | 목차의 근거 |
| bulleted / numbered list | 목록 | 중첩 허용 |
| to-do (checkbox) | 체크리스트 | 위키에서 빈번 |
| code block | 코드 | 개발 위키 필수, 언어 태그 |
| quote / divider | 강조·구분 | 가벼움 |

**제외(2차):** 테이블, 이미지/파일 업로드, 임베드, 토글, 컬럼 레이아웃, 데이터베이스 뷰. → MLP는 "텍스트 위키"에 집중, CRDT 수렴 증명 범위를 통제.

> ⚠️ 블록 타입을 늘릴수록 CRDT 동시편집 수렴 검증 표면이 커진다. MLP 세트는 **Yjs가 표준 지원하는 범위**라 M1 증명을 그대로 확장한다(신규 수렴 리스크 최소).

## 3. 핵심 사용자 여정 (User Journeys)

### J1. 워크스페이스 시작 → 첫 페이지 (온보딩)
```
가입/로그인(JWT)
  → 워크스페이스 자동 생성(또는 초대 수락)
  → 빈 상태: "첫 페이지 만들기" CTA
  → 페이지 생성 → 제목 입력 → 블록 작성 시작
```
> 빈 상태(empty state)는 MLP 필수. 아무것도 없을 때 막히면 "써지는 제품"이 아니다.

### J2. 동료 초대 → 동시 편집 (핵심 데모 시나리오)
```
워크스페이스 멤버 초대(email)
  → 동료 수락 → 같은 페이지 진입
  → 두 커서가 같은 문단을 동시에 편집 → 충돌 없이 수렴   ★M1에서 증명됨
  → (M3) 서로의 커서·선택 영역이 실시간 표시
```

### J3. 페이지 구조화 (트리 조작)
```
새 페이지를 특정 페이지 "안에" 생성  → parent_id 설정
사이드바에서 페이지를 드래그해 다른 위치로 이동 → reparent
  → doc-service가 사이클 검사 + 직렬화 (CRDT 아님, §4 데이터문서)
페이지 아카이브(삭제) → 트리에서 숨김, 스냅샷은 보존
```

### J4. 페이지 공유 (권한)
```
페이지 "공유" → 멤버 선택 → editor/viewer 지정
  → 하위 페이지로 권한 상속 (가장 가까운 명시 권한 우선)
viewer로 들어온 사람: 읽기만, 편집 차단(게이트웨이 강제)
```

### J5. (M4) AI co-pilot — 본 문서 범위 밖, 자리만 표시
```
페이지에서 "이 문서에 X 관련 뭐라 했지?" → AI가 문서 근거로 답(SSE 스트리밍)
선택 문단 요약/리라이트 → 결과는 사용자가 명시적으로 삽입
```
> AI UX 상세 플로우는 우선순위에서 제외됨 → M4 착수 시 별도 보강. PRD §4.2가 현재 범위 정의의 권위.

## 4. 반드시 정의해야 할 상태 (States) — 기존 기획의 가장 큰 공백

"써지는 제품"과 데모의 차이는 **예외 상태 처리**에서 갈린다. MLP가 정의해야 할 최소 상태:

| 상황 | 정의 필요 | MLP 처리 |
|---|---|---|
| 빈 워크스페이스/페이지 | empty state | "첫 페이지/첫 블록" 안내 + CTA |
| 연결 끊김(WS) | 편집 중 네트워크 단절 | 로컬 편집 유지 + 재연결 시 재동기화(Yjs 특성) + 상태 배지 |
| 권한 없음 | 링크로 접근했으나 권한 X | 403 안내 화면(요청 보내기는 2차) |
| 동시 이동 충돌 | 트리 reparent 경합 | 마지막 이동 우선, 사이클이면 거부+토스트 |
| AI 실패/타임아웃 | (M4) 추론 실패 | error 이벤트 + 폴백/재시도 안내 |
| 저장 실패 | 스냅샷 영속 실패 | (M2) 재시도 + 사용자 경고 |

> 이 표가 기존 PRD에 통째로 빠져 있었다. MLP 정의의 핵심.

## 5. MLP 컷 라인 (이 문서 + 데이터문서 종합)

```
┌─ MLP (실제 써지는 최소) ────────────────────────────────┐
│ 워크스페이스 1 · 멤버 초대 · 페이지 트리(생성/이동/삭제) │
│ 블록 에디터(7개 타입) · 실시간 수렴 · 영속화            │
│ 권한 2계층(워크스페이스+페이지 오버라이드, editor/viewer)│
│ 핵심 상태(빈 상태·연결끊김·권한없음)                    │
└────────────────────────────────────────────────────────┘
        ↑ 여기까지가 "사내 위키로 실제 사용 가능"
┌─ 확장 (2차) ────────────────────────────────────────────┐
│ AI co-pilot(M4) · presence 커서(M3) · 트리 CRDT         │
│ 고급 블록(테이블/이미지/임베드) · 버전 히스토리          │
│ 공개 링크 · 외부 게스트 · 멀티 워크스페이스 UI · 검색     │
└────────────────────────────────────────────────────────┘
```

> 이 컷 라인은 기존 마일스톤(M2 영속·권한 / M3 presence / M4 AI)과 충돌하지 않는다. MLP의 "권한·트리"는 **M2 doc-service 범위**로 들어간다.

## 2. 만드는 이유 (포트폴리오 관점)

이 프로젝트는 **사업이 아니라 역량 증명**이 1차 목표다. 증명하려는 것:

1. **백엔드 애플리케이션 로직 깊이** — 분산 동시성(CRDT)이라는 비자명한 문제를 직접 다룬다. (플랫폼/SRE 색채와 차별화)
2. **언어를 도구로 선택하는 판단력** — Java/Python/Rust를 각 서비스 요구사항에 맞춰 분리한다. "다양하게 써봄"이 아니라 "필연적 분리"임을 설명할 수 있어야 한다.
3. **AI 통합을 리소스 제약 하의 백엔드 설계 문제로 다룸** — 단순 API 호출이 아니라, 단일 GPU 한계 위에서의 추론 큐잉·백프레셔·폴백.
4. **폴리글랏 관찰성** — Java→Rust→Python 호출 체인을 단일 OpenTelemetry trace로 잇는다.

## 3. 타겟 사용자 / 핵심 유스케이스

**1차 사용 맥락(2026-06-30 확정): 사내 위키/지식베이스.** 소규모 팀이 워크스페이스 안에서 페이지 트리로 지식을 지속적으로 쌓고, 같은 페이지를 실시간으로 함께 편집한다. (제품 상세는 [3. 정보구조·UX](3-information-architecture-ux.md), 모델은 [4. 데이터·권한](4-data-and-permission-model.md))

| 페르소나 | 맥락 | 핵심 시나리오 |
|---|---|---|
| **워크스페이스 운영자** (팀 리드) | 위키 구조·멤버·권한 관리 | 페이지 트리를 설계하고, 동료를 초대하고, 민감 페이지는 viewer로 제한 |
| **협업 작성자** (팀원) | 일상적으로 문서를 쓰고 고침 | 회의록·런북을 동료와 **같은 페이지에서 동시 편집**, 서로의 커서를 본다(M3) |
| **지식 질의자** (신규/타팀) | 쌓인 위키에서 답을 찾음 | "배포 절차 어디 있지?"를 **AI에게 문서 근거로 질의**(M4), 트리·검색으로 탐색 |
| **편집자** | 초안을 다듬음 | 선택 문단을 AI에게 요약/리라이트 요청, 결과 검토 후 **명시적으로 삽입**(M4) |

> 위키 맥락이라 **"누구나 읽고 고치되, 일부는 제한"** 이 자연스럽다 → 워크스페이스 멤버십 baseline + 페이지별 오버라이드 권한 모델([4 §4](4-data-and-permission-model.md)).

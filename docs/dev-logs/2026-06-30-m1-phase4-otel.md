# M1 Phase 4 — 폴리글랏 OTel trace 2-hop (Java→Rust 단일 trace) 실측

가드레일 4("서비스 간 호출은 gRPC + OTel propagator 통과")를 **실측으로 증명**한 기록.
PRD §7 DoD #6("Java→Rust→Python 단일 trace")의 **2-hop 부분**(Java gateway → Rust engine) 달성.
3-hop(+Python ai-service)은 M4. 이 dev-log는 4.1(engine)·4.2(gateway)·4.3(live)을 합본 후향 기록한다
(PDD-01: Phase 1~3는 dev-log 있으나 4.x 누락이었음 → 해소).

## 무엇을 만들었나 — thin 2-hop

```
브라우저(Yjs) ──WS──▶ ws-gateway(Java) ──gRPC bidi Sync──▶ crdt-engine(Rust)
                         │ javaagent가 stream-open HTTP/2 헤더에         │ MetadataExtractor가
                         │ W3C traceparent 주입                          │ traceparent 추출→span.set_parent
                         ▼                                               ▼
                     [ws-gateway span] ◀──── 같은 traceID ────▶ [wedocs-crdt-engine span]
```

- **Java(gateway)** = OTel **javaagent**(런타임 부착, **앱 코드 0**) — grpc-java 클라 자동계측이 traceparent out-of-box 주입.
- **Rust(engine)** = 수동 추출(`global::get_text_map_propagator().extract(MetadataExtractor)` → `set_parent` → `.instrument`) + `opentelemetry-otlp`(gRPC).
- **정직한 한계**: traceparent는 bidi 스트림 **open 시 1회**(HTTP/2 헤더) 주입 → 산출물은 "한 *편집*"이 아니라 **"한 WS 세션(스트림 open)이 Java span→Rust span 단일 trace"**. per-edit 메시지 span = M5(proto field 또는 수동 컨텍스트 전파 필요).

## ★ 교훈 1 — config-contract-audit 실증: javaagent 기본 OTLP 프로토콜 변경

`OTEL_EXPORTER_OTLP_ENDPOINT=http://localhost:4317`(gRPC 포트)로 줬는데 gateway export가 전부 실패:
`java.io.IOException: unexpected end of stream on http://localhost:4317/`. 원인 = **OTel Java agent 2.x의 기본 OTLP
프로토콜이 `http/protobuf`로 바뀜** → HTTP를 gRPC 포트(4317)에 보냄. engine(Rust)은 gRPC라 정상 도착, gateway만 실패.

→ `OTEL_EXPORTER_OTLP_PROTOCOL=grpc`로 engine과 통일(c8374d5). 수정 후 export 에러 0, 단일 trace 성립.

이것은 `config-contract-audit.md`가 경고한 **"OTel 기본값이 그대로일 것 → exporter protocol 변경, traces drop"**
시나리오 그 자체다. **컴파일·부팅 스모크로는 안 잡히고 live export에서만 드러남** → 4.3 live 검증의 존재 이유.

## ★ 교훈 2 — M1R-09: "고치기 전에 spec 검증"이 회귀를 막음

감사(plan-audit)에서 리뷰어가 "`telemetry.rs`가 endpoint를 exporter에 안 넘김 → `.with_endpoint()` 추가" 권장.
docs.rs 0.32 검증 결과: `with_tonic().build()`가 `OTEL_EXPORTER_OTLP_ENDPOINT`(+signal-specific `_TRACES_ENDPOINT`
우선)를 **자동으로 읽음** → 현 코드 기능 정상. **권장대로 `.with_endpoint()`를 넣었으면 signal-specific override를
덮어쓰는 회귀**가 됐을 것. live에서 engine 로그 `exporter=otlp/grpc → http://localhost:4317` + engine span 도착으로
자동읽기 실증. → **engine 코드 무변경**(deep-thinking/workflow "추측 전 spec 검증"이 회귀를 막은 사례).

## 교훈 3 — bidi 스트림 client span은 스트림 종료 시 export

처음 Jaeger 조회 시 `wedocs-crdt-engine` span만 있고 `ws-gateway` span은 없었다(engine span의 parent가 dangling).
gRPC **bidi 스트리밍 client span은 스트림이 닫혀야 종료·export**되기 때문. E2E가 연결을 닫으면(teardown) gateway
스트림 종료 → span export → batch flush(`OTEL_BSP_SCHEDULE_DELAY=1000`) 후 두 span이 같은 traceID로 합류.

## 재현 노트 (fresh 머신)

- engine: `make proto-sync`(buf export vendoring) **선행** — 없으면 build.rs가 `proto/crdt/crdt.proto` 못 찾아 panic.
- PATH: 백그라운드/비대화 셸은 `~/.cargo/bin`(cargo)·`~/.local/bin`(buf) 미포함 → 명시 export.
- gateway: `make run-otel`이 jar 자동 다운로드 + bootJar + `java -javaagent`. `OTEL_EXPORTER_OTLP_PROTOCOL=grpc` 필수.
- 버전: opentelemetry-otlp 0.32 · javaagent 2.29.0 · Jaeger v2 2.19.0.

# M1.5 — CRDT 엔진 벤치 Tier1 위생 수정 (engine PR #3 머지)

[벤치마크 방법론](../design/benchmark-methodology.md) §9 로드맵의 "즉시(M1.5)" 항목
— **Tier1 머지 마이크로벤치의 측정 위생 수정 + 그룹화 + 회귀 가드** — 을 구현해 머지한 기록.
코드는 서비스 레포 `weDocs-crdt-engine` ([PR #3](https://github.com/ressKim-io/weDocs-crdt-engine/pull/3), 머지 `0b97c4a`).

## 무엇이 문제였나 (방법론 §0)

기존 `benches/convergence.rs` 단일 케이스 `apply_256_concurrent_updates`:
- `Doc::new()` + `Update::decode_v1` 가 **`b.iter()` 안**에 있어 alloc·decode 비용이 머지 측정에 혼입(§7 위반).
- 케이스 1개("256개 1글자 독립 삽입")뿐 → 순차 타이핑·삭제·붙여넣기 없음(§3 비대표성).
- baseline 비교 장치 없음 → 회귀 추적 불가(§4).

## 무엇을 했나

1. **측정 위생(§7)** — `iter_batched` 로 `Doc::new()`+`decode_all()` 을 setup(타이밍 밖)으로 분리.
   - `merge` 그룹 = 순수 apply(머지 핫패스)만 측정.
   - `build_doc` 그룹 = 기존 전체 파이프라인(alloc+decode+merge). **둘의 델타 = decode+alloc 비용.**
   - per-update 1 트랜잭션 유지 → 실서비스 `service.rs::apply_v1`(프레임당 1 apply) 경로 대표성.
2. **워크로드 4종(§3)** — `sequential_typing`(같은 client diff) / `concurrent_inserts`(기존) /
   `with_deletes`(tombstone, `remove_range`) / `large_paste_10k`(단일 큰 update).
3. **회귀 가드(§4)** — `make bench-baseline`(`--save-baseline main`) / `make bench-compare`(`--baseline main`).
   엔진 레포에 CI(`.github/workflows`)가 없어 **Makefile 타깃 + README 사용법**으로 도입(CI 파이프라인 신설은 범위 밖).
4. `throughput(ElementsAndBytes)` → ops/sec + MiB/s 동시 보고(§7).

## 실측 — 정식 baseline (방법론 §7·§8 기록 규약)

**측정 환경**: commit `0b97c4a` · AMD Ryzen 5 7500F(6C/12T) · Linux 7.0.0-27 x86_64 ·
rustc 1.96.0 · criterion 0.8.2 기본 config(100 samples / warm-up 3s / measure 5s) · loadavg 0.00(idle).
⚠️ CPU governor=`powersave`(performance 아님, sudo 미변경) → 절대치 ±2% 변동 가능 → **merge↔build_doc 델타**에 주목.
criterion `main` baseline 은 `target/criterion/`(gitignored) 로컬 회귀 기준; 아래 표가 이식 가능한 기록본.

| case (median) | `merge`(apply only) | `build_doc`(+decode+alloc) | decode+alloc 델타 |
|---|---|---|---|
| sequential_typing_256 | 123.5 µs (49.4 MiB/s) | 146.7 µs | ~23 µs |
| concurrent_inserts_256 | 860.5 µs (5.65 MiB/s) | 900.1 µs | ~40 µs |
| with_deletes_50pct | 174.7 µs (44.0 MiB/s) | 207.4 µs | ~33 µs |
| large_paste_10k | **682.6 ns** (13.7 GiB/s) | **4.871 µs** | **~4.19 µs** |

두 가지 정직한 관찰:
- **large_paste**: decode+alloc(~4.2µs)이 순수 apply(0.68µs)의 **약 6배**. alloc·decode를 머지 시간에 함께 재던
  기존 벤치의 숫자는 머지 성능이 아니라 디코딩 비용까지 섞인 값이었던 셈 — "정직하지 않은 N배"가 시작되는 지점.
- **concurrent_inserts(860µs)가 sequential_typing(123µs)의 ~7배 느림** — 둘 다 256개 1글자 update지만,
  256개 독립 client가 같은 origin에 동시 삽입하면 yrs 충돌 통합이 거의 quadratic, 같은 client 순차 append는 linear.
  워크로드를 나눈 이유가 수치로 드러남(Tier2 샤딩 "N배"와는 별개 축).

이후 PR은 `make bench-compare`(`--baseline main`)로 이 기준 대비 회귀를 감지한다.

## 검증

`cargo fmt --check` · `cargo clippy --benches -D warnings` · `cargo bench --bench convergence`(8케이스) ·
`--save-baseline main` → `--baseline main` 비교 동작 확인. 수렴 정확성은 기존 `tests/convergence_proptest.rs` 가 계속 가드.

## 교훈 / 주의

- **케이스 id 변경**(`apply_256_concurrent_updates` → `merge/concurrent_inserts_256`)으로 기존 criterion 기록과
  직접 비교 불가 → `--save-baseline main` 으로 새 기준 수립.
- `cargo bench` 는 lib 단위테스트(libtest harness)도 함께 돌아 criterion CLI 플래그를 거부 →
  criterion 전용 플래그는 **`cargo bench --bench convergence -- <flags>`** 로 타깃 지정.
- 검증 전 criterion 0.8.2 / yrs 0.27.2 API 를 docs.rs·캐시 소스로 사전 확인(추측 코딩 회피, `deep-thinking`).
